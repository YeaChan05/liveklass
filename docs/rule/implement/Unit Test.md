# Unit Test

단위 테스트는 도메인 규칙과 서비스 로직을 빠르게 검증하기 위한 테스트다

## 목적

- 하나의 기능 단위를 독립적으로 검증한다
- 외부 인프라 없이 빠르게 실행되어야 한다
- 비즈니스 규칙의 성공/실패 조건을 명확히 드러낸다

## 작성 대상

다음 코드는 단위 테스트를 우선 작성한다

- Domain Model
- Value Object
- Service
- UseCase
- Policy
- Validator
- Mapper

## 작성 규칙

- 테스트 대상의 책임만 검증한다
- DB, Redis, MQ, 외부 API에 의존하지 않는다
- 인프라 검증이 필요하면 통합 테스트로 분리한다
- 하나의 테스트는 하나의 행위를 검증한다
- 테스트 이름은 조건, 행위, 기대 결과가 드러나야 한다

## TDD 규칙

기능 구현 전에 실패하는 단위 테스트를 먼저 작성한다

흐름은 반드시 다음 순서를 따른다

1. 요구사항을 테스트 케이스로 변환한다
2. 실패하는 단위 테스트를 작성한다
3. RED를 확인한다
4. 최소 구현으로 테스트를 통과시킨다
5. GREEN을 확인하고 RED인 경우 구현을 수정한다
6. 가능하면 리팩토링한다

## 금지 사항

- 구현 세부사항을 과하게 검증하지 않는다
- 단순 getter/setter 테스트를 작성하지 않는다
- mock을 남용해 실제 행위를 숨기지 않는다
- 통과만을 위한 의미 없는 assertion을 작성하지 않는다
- 기존 테스트 스타일과 다른 형식을 임의로 도입하지 않는다

## 코드 작성 방식
- JUnit 5와 AssertJ를 사용한다
- 단위 테스트 클래스는 `*Test`로 끝나야 한다
- AAA(Arrange/Act/Assert) 패턴을 따르며 주석으로 구분한다
