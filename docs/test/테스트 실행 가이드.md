# 테스트 실행 가이드

이 문서는 liveklass의 테스트 실행 명령과 k6 정합성 시나리오 실행 방법을 정리합니다.

## 기본 검증

전체 테스트와 Jacoco 커버리지 검증을 실행합니다.

```bash
./gradlew test
```

전체 Jacoco 리포트만 다시 생성할 때는 아래 명령을 사용합니다.

```bash
./gradlew jacocoRootReport
```

커버리지 결과는 `build/reports/jacoco/jacocoRootReport/html/index.html`에서 확인합니다.

## 모듈별 테스트

단위 테스트는 각 모듈의 `test` 태스크로 실행합니다.

```bash
./gradlew :course:service:test
./gradlew :course:api:test
./gradlew :course:model:test
```

통합 테스트는 `src/integrationTest`가 있는 모듈에만 생성됩니다. Redis, MySQL 기반 통합 테스트는 Testcontainers를 사용하므로 Docker가 실행 중이어야 합니다.

```bash
./gradlew :course:repository-redis:integrationTest
./gradlew :course:repository-jpa:integrationTest
./gradlew :course:repository-jdbc:integrationTest
```

코드 스타일은 ktlint로 확인합니다.

```bash
./gradlew ktlintCheck
```

## E2E 정합성 테스트

k6 시나리오는 실행 중인 애플리케이션을 대상으로 수강 신청 정합성을 검증합니다.

먼저 토큰 생성과 결제 대기 만료 시간을 설정해 애플리케이션을 실행합니다.

```bash
MEMBER_TOKEN_GENERATOR_ENABLED=true \
MEMBER_TOKEN_GENERATOR_COUNT=10000 \
MEMBER_TOKEN_GENERATOR_OUTPUT_PATH=./k6/tokens.json \
AUTH_TOKEN_ACCESS_EXPIRES_IN=14400 \
ENROLLMENT_PAYMENT_PENDING_EXPIRES_IN=60s \
ENROLLMENT_PAYMENT_EXPIRATION_FIXED_DELAY_MS=5000 \
./gradlew :course:application:bootRun
```

애플리케이션이 기동된 뒤 k6가 실행 가능한 위치에서 아래 시나리오 중 필요한 것을 실행합니다.


## E2E 실행 주의사항

- 정합성 검증 테스트 진행 시 `MEMBER_TOKEN_GENERATOR_ENABLED=true`로 토큰 생성 기능을 활성화합니다.
- k6 토큰은 애플리케이션 기동 시 생성됩니다.
- 오래된 `k6/tokens.json` 재사용은 지양합니다.
- 큰 시나리오 실행 시 `MEMBER_TOKEN_GENERATOR_COUNT`를 `APPLICANT_COUNT` 이상으로 설정합니다.
- 1만 명 시나리오 실행 시 `MEMBER_TOKEN_GENERATOR_COUNT=10000`이 필요합니다.
- 토큰 만료 방지를 위해 `AUTH_TOKEN_ACCESS_EXPIRES_IN`을 충분히 길게 설정합니다.
- 결제 대기 만료 기반 시나리오 실행 시 `ENROLLMENT_PAYMENT_PENDING_EXPIRES_IN`과 k6의 대기 시간 값을 맞춥니다.
- `ENROLLMENT_PAYMENT_PENDING_EXPIRES_IN=60s`, `ENROLLMENT_PAYMENT_EXPIRATION_FIXED_DELAY_MS=5000` 기준 `WAIT_BEFORE_REFILL_SECONDS=70`을 권장합니다.
- `WAIT_BEFORE_REFILL_SECONDS`가 너무 짧으면 대기자가 계속 `WAITLISTED` 상태로 남아 실패할 수 있습니다.
- `WAIT_BEFORE_REFILL_SECONDS`가 너무 길면 승격된 `PENDING` 신청이 다시 만료될 수 있습니다.
- 결제 대기 만료 기반 시나리오는 이전 테스트 데이터가 남아 있으면 로그 해석이 어려울 수 있으므로 필요 시 DB 초기화 후 실행합니다.
- waitlist Redis key TTL은 기본 1일이며 `ENROLLMENT_WAITLIST_TTL`로 조정할 수 있습니다.
- 로컬 로그는 `./logs`에 파일로 저장되고 Promtail이 Loki로 수집합니다.
- Grafana의 `Spring Boot Observability` 대시보드에서 `log_keyword`를 입력하면 로그 패널과 `compose_service=~"app-.*"` 기준 로그 메트릭을 확인할 수 있습니다.
